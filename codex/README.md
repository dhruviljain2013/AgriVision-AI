# AgriVision AI - Intelligent Plant Health Assistant

AgriVision AI is an original CBSE science project that will use a custom-trained image classification model to identify whether a plant leaf is:

- Healthy
- Diseased

The project will be trained only on approved datasets imported into this project. The current setup includes project architecture, dataset management, dataset importing, a TensorFlow training pipeline, backend prediction utilities, evaluation utilities, and a disease knowledge base. It does not include a GUI.

## Project Goals

- Build a simple and explainable plant health assistant.
- Train a binary image classifier using personally collected plant leaf images.
- Keep the project organized so each stage can be developed safely and tested separately.
- Make the final application suitable for presentation as a CBSE science project.

## Current Phase

Phase 4 Final: Professional plant disease AI backend foundation.

The project can now import existing plant disease datasets, prepare images, train a configurable transfer learning model, evaluate results, and prepare backend prediction responses. No GUI has been added yet.

## Purpose of Each Folder and File

### Root Files

- `README.md`: Explains the project idea, goals, structure, and development phases.
- `requirements.txt`: Lists Python libraries expected to be used in later phases.
- `.gitignore`: Prevents temporary files, virtual environments, datasets, trained models, and generated outputs from being committed accidentally.

### `config/`

Stores configuration files such as image size, class names, split ratios, and file paths. Keeping settings here makes the project easier to adjust without changing Python code.

- `config/dataset_config.json`: Controls Phase 2 dataset preparation.
- `config/training_config.json`: Controls Phase 3 model training, output paths, batch size, epochs, callbacks, and learning rate.
- `config/disease_database.json`: Stores disease descriptions, causes, symptoms, treatment, and prevention details.

### `data/`

Stores project datasets. Large datasets should stay local and should not be uploaded to Git unless there is a clear reason.

- `data/raw/`: Original collected leaf images.
- `data/raw/healthy/`: Original images of healthy leaves.
- `data/raw/diseased/`: Original images of diseased leaves.
- `data/processed/`: Cleaned, resized, renamed, and split dataset files generated by the dataset manager.
- `data/processed/standardized/`: Valid images converted to RGB, resized to the configured size, and renamed clearly.
- `data/processed/training/`: Images used later to train the model.
- `data/processed/validation/`: Images used later to tune and check the model during development.
- `data/processed/testing/`: Images kept aside for final testing after training.
- `data/samples/`: Small example images that may be useful for documentation or presentation.

### `docs/`

Stores project explanation material, science fair notes, methodology, observations, diagrams, and presentation planning.

### `models/`

Stores future trained model files. These files can become large, so they are ignored by Git by default.

### `notebooks/`

Stores future Jupyter notebooks for experiments, visual checks, dataset exploration, and result analysis.

### `reports/`

Stores generated project outputs such as accuracy charts, confusion matrices, training graphs, and final result summaries.

- `reports/figures/`: Future image-based results and charts.
- `reports/training/`: Saved training history.
- `reports/evaluation/`: Saved evaluation metrics.
- `reports/tensorboard/`: TensorBoard logs generated during training.

### `src/agrivision_ai/`

Contains the main Python source code. Keeping source code inside a package folder makes the project cleaner and easier to test.

- `src/agrivision_ai/__init__.py`: Marks the folder as a Python package.
- `src/agrivision_ai/dataset_importer.py`: Imports PlantVillage, PlantDoc, and similar datasets from ZIP files or folders.
- `src/agrivision_ai/dataset_manager.py`: Phase 2 dataset management script.
- `src/agrivision_ai/disease_database.py`: Loads and searches the disease knowledge base.
- `src/agrivision_ai/evaluation_utils.py`: Reusable accuracy, precision, recall, F1, confusion matrix, and per-class accuracy utilities.
- `src/agrivision_ai/training_config.py`: Loads and validates training configuration.

## Phase 2 Dataset Management

The dataset manager prepares your own collected leaf images before any AI model is trained.

Place your images here:

```text
data/raw/healthy/
data/raw/diseased/
```

Then run:

```bash
python src/agrivision_ai/dataset_manager.py
```

The script uses settings from:

```text
config/dataset_config.json
```

### What the Dataset Manager Does

1. Validates every image:
   - Opens each image safely.
   - Detects corrupted images.
   - Ignores unsupported file formats.

2. Renames images automatically:
   - Healthy images become names like `healthy_0001.jpg`.
   - Diseased images become names like `diseased_0001.jpg`.

3. Resizes every valid image:
   - Default size is `224x224`.
   - This can be changed in `config/dataset_config.json`.

4. Converts every image to RGB:
   - This gives all images the same color format.
   - Consistent image format is important because machine learning models expect uniform input.

5. Generates dataset statistics:
   - Total images.
   - Healthy images.
   - Diseased images.
   - Original image sizes.
   - Average original resolution.
   - Number of corrupted images.
   - Number of unsupported files ignored.

6. Splits the dataset:
   - `training`
   - `validation`
   - `testing`

The default split is:

```text
training: 70%
validation: 15%
testing: 15%
```

You can change these values in `config/dataset_config.json`, but they must add up to `1.0`.

### Why Splitting the Dataset Matters

The training set teaches the model. The validation set helps check progress while developing. The testing set is kept separate so the final result is more honest. If the same images are used for everything, the model may look better than it really is.

### Generated Output

After the script runs, it creates:

```text
data/processed/standardized/
data/processed/training/
data/processed/validation/
data/processed/testing/
data/processed/dataset_statistics.json
```

The generated image files and statistics report are ignored by Git so large datasets are not committed accidentally.

## Dataset Importer

To import an existing dataset such as PlantVillage or PlantDoc:

```bash
python src/agrivision_ai/dataset_importer.py path/to/dataset.zip
```

or:

```bash
python src/agrivision_ai/dataset_importer.py path/to/extracted_dataset_folder
```

The importer detects class folders such as `Tomato___Healthy`, `Tomato___Early_Blight`, and `Apple___Scab`, then organizes valid images into:

```text
data/raw/healthy/
data/raw/diseased/
```

It creates:

```text
data/raw/metadata.csv
reports/dataset_import_report.json
```

## Phase 3 AI Training Pipeline

The training pipeline uses TensorFlow/Keras and MobileNetV2 transfer learning.

The current training config loads the already-extracted dataset directly from:

```text
C:/Users/oswal/Downloads/LeafDataset/image data/train
C:/Users/oswal/Downloads/LeafDataset/image data/validation
C:/Users/oswal/Downloads/LeafDataset/image data/test
```

Then run:

```bash
py -3.11 src/agrivision_ai/train.py
```

The script uses settings from:

```text
config/training_config.json
config/dataset_config.json
```

### What the Training Pipeline Does

1. Loads image batches:
   - Images are loaded directly from the extracted training, validation, and testing folders.
   - No images are copied or duplicated into `data/raw` or `data/processed`.
   - Batch loading means the model sees small groups of images at a time instead of loading everything into memory at once.

2. Applies data augmentation:
   - Random horizontal flips.
   - Small rotations.
   - Small zoom changes.
   - Small contrast changes.

Data augmentation helps the model handle natural photo differences, such as leaf angle, camera distance, and lighting.

3. Uses MobileNetV2 transfer learning:
   - MobileNetV2 is the default model architecture.
   - EfficientNetB0 and ResNet50 are optional through `config/training_config.json`.
   - The selected model is loaded with ImageNet weights.
   - Its original layers are frozen and used only as a feature extractor.
   - A new classifier head is trained only on your configured plant/disease leaf classes.

4. Uses training callbacks:
   - `EarlyStopping`: Stops training when validation loss stops improving.
   - `ModelCheckpoint`: Saves the best model to `models/best_model.keras`.
   - `ReduceLROnPlateau`: Lowers the learning rate when training progress slows.
   - `TensorBoard`: Saves training logs for visual inspection.

5. Saves training outputs:
   - Best model: `models/best_model.keras`
   - Last model: `models/last_model.keras`
   - Training history: `reports/training_history.json`
   - Accuracy graph: `reports/figures/accuracy_graph.png`
   - Loss graph: `reports/figures/loss_graph.png`

6. Evaluates the trained model:
   - Accuracy.
   - Precision.
   - Recall.
   - F1 Score.
   - Confusion Matrix.

Evaluation outputs are saved here:

```text
reports/classification_report.json
reports/figures/confusion_matrix.png
```

### What the Metrics Mean

- Accuracy: Overall percentage of correct predictions.
- Precision: Out of the images predicted as Diseased, how many were actually Diseased.
- Recall: Out of the actual Diseased images, how many the model found.
- F1 Score: A balanced score that combines precision and recall.
- Confusion Matrix: A table showing correct and incorrect predictions for each class.

### Important Notes

- The model trains only on the extracted dataset configured in `config/training_config.json`.
- MobileNetV2 ImageNet weights are used only as a general feature extractor.
- The first training run may need internet access so TensorFlow can download the MobileNetV2 ImageNet weights.
- For honest results, do not place the same leaf photo in training, validation, and testing.

## Suggested Architecture Improvements Before Future Phases

Before writing the next phases, the following decisions should be made:

1. Confirm image collection rules:
   - Same camera or phone if possible.
   - Clear leaf images with good lighting.
   - Avoid mixing many unrelated plant species unless that is intentional.

2. Decide whether the dataset will contain one plant type or multiple plant types:
   - One plant type is simpler and better for a CBSE project.
   - Multiple plant types need more images and careful explanation.

3. Decide the target interface:
   - Simple command-line prediction.
   - Streamlit web app.
   - Tkinter desktop app.

4. Keep the first model simple:
   - Start with transfer learning or a small CNN.
   - Focus on explaining the science clearly instead of making the system unnecessarily complex.

5. Plan dataset splitting early:
   - Training set.
   - Validation set.
   - Test set.

6. Track limitations honestly:
   - The assistant should support observation, not replace expert agricultural advice.
   - Results will depend heavily on image quality and dataset size.
